# Civic Economy Doctrine (2026-04-07)

## Purpose
Define the minimum governing doctrine for ANU’s economy-adjacent functionality before any broader financial expansion.

This doctrine is designed to prevent “payment features” from outrunning legitimacy, disclosure, or trust posture.

## Programme Rule
Economy work in this phase is:
- doctrine first,
- disclosure first,
- audit first,
- pilot only beyond currently proven functionality.

## Canonical Source-of-Truth Model
### Default assumption
The backend emits the canonical public finance disclosure read model.
The impact-service remains the computation and approval engine.

### Consequence
- public trust, sponsor, and disclosure surfaces read from backend canonical disclosure models,
- impact-service remains authoritative for internal event/approval computation but not for final public narrative shaping.

This remains subject to decision D002.

## Economic Object Classes
### Core object classes for this phase
- pool
- disbursement
- sponsorship disclosure
- trust report linkage
- approval record
- revenue event summary
- allocation summary

### Deferred classes
- generalized marketplace behaviour,
- broad commerce rails,
- user-level payment expansion,
- federation-wide fund flow.

## Non-Distortion Rule
Sponsor or finance data must never alter:
- base discovery order,
- archive order,
- governance order,
- model-registry order,
- connector destination order.

This rule is mandatory and not optional.

## Public Disclosure Doctrine
Public-facing economy surfaces must answer:
1. what exists,
2. what is sponsored,
3. what rules apply,
4. what has been approved or disbursed,
5. what is withheld and why,
6. what source generated the disclosure.

## Placement Doctrine
Sponsor disclosures may appear on:
- trust surfaces,
- transparency surfaces,
- archive-linked records,
- clearly labelled public trust modules.

Sponsor disclosures may **not** silently appear as:
- hidden weighting,
- disguised “featured” ordering without disclosure,
- altered archive or governance sequencing.

## Approval Doctrine
No consequential economy event is complete unless it has:
- an actor,
- a scope,
- an approval posture,
- an audit trail,
- a public disclosure rule.

Where public disclosure is not appropriate, the trust surface must still explain that restricted treatment exists.

## Backend Responsibilities
The backend is responsible for:
- canonical public disclosure shaping,
- public trust report linkage,
- public sponsor disclosure payload,
- archive linkage for finance-adjacent records.

## Impact-Service Responsibilities
Impact-service is responsible for:
- approval/event computation,
- allocation/disbursement primitives,
- underlying impact-side financial state transitions,
- exposing validated projection data to backend.

## Frozen Areas
The following remain frozen in this phase:
1. broad payment feature expansion,
2. sponsor-influenced ordering of public surfaces,
3. generalized donations/subscriptions beyond current proven rails unless doctrine and disclosures are already wired,
4. cross-node pooled economy without node binding and isolation proof.

## Minimum Required Artifacts for M4
1. sponsor disclosure payload,
2. public trust report model/API,
3. non-distortion tests,
4. one public disclosure surface,
5. one archive or trust record referencing sponsor/economy context,
6. approval audit proof.

## Invalid States
The following are unacceptable:
- public sponsor influence with no disclosure,
- economy UI with no doctrine,
- multiple public finance truths across services,
- disbursement narrative with no audit trail,
- hidden sponsor placement in discovery or governance surfaces.
