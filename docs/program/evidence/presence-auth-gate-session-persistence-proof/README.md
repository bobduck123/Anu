# Presence Owner Auth Gate and Session Persistence Proof

## Summary

This pass fixes the remaining Canvas Builder V2 owner-route blocker: the browser could observe an empty Supabase session once during sign-in or route hydration, immediately render the owner access gate, and never recover without another user action.

The repair is authentication-only:

- password sign-in waits until the browser client can read back its persisted session before redirecting;
- owner editor and private preview use the same hydration-aware session resolver;
- route gates distinguish access checking, confirmed denial, and a retryable inability to confirm access;
- the diagnostic query flag remains display-only and is tested not to bypass owner access.

Local auth, Canvas lifecycle, public boundary, publish confirmation, and RoomKey regression tests pass. The corrected source has not yet been deployed and re-smoked with the real hosted owner account.

## Root Cause Found

`useOwnerNode()` and the private preview each performed a one-shot `supabase.auth.getSession()` read. If that initial browser-side read returned no session while Supabase cookie/session persistence was still settling, the application converted a transient state into `Sign in required` and rendered the same gate as a genuinely anonymous user.

The sign-in form also redirected immediately after `signInWithPassword()` returned, without verifying that the session was readable by the owner-route guard.

## Why Debug Worked While Standard Appeared Gated

Source inspection found no authorization branch controlled by `?debug=1`. Both `/studio/[id]/editor` and `/studio/[id]/editor?debug=1` enter the same route component and the same `useOwnerNode()` guard. The debug flag is consumed only inside an already rendered editor to show diagnostics.

The safe conclusion is that the hosted observation was timing/state dependent: the debug navigation occurred after session/backend state had become readable, while the earlier standard navigation had already stored a terminal gate result. This pass removes that sticky one-shot behavior and adds explicit parity tests.

## One-Login Persistence Fix

- [ownerSession.ts](/C:/Dev/Flora_fauna/presence-app/components/studio/ownerSession.ts) now performs a bounded session hydration check rather than accepting an immediate null session as definitive.
- [PresenceAuthForms.tsx](/C:/Dev/Flora_fauna/presence-app/components/auth/PresenceAuthForms.tsx) waits for the persisted owner session before navigating into Studio.
- [useOwnerNode.ts](/C:/Dev/Flora_fauna/presence-app/components/studio/useOwnerNode.ts) exposes explicit access state and only emits sign-in/forbidden after confirmation.
- A retryable owner-read failure now presents `Unable to confirm access` with `Try again`, not a false owner denial.

## Preview Route Fix

[PresenceDraftPreviewPage.tsx](/C:/Dev/Flora_fauna/presence-app/components/studio/editor/PresenceDraftPreviewPage.tsx) now uses the same hydration-aware owner session resolution as the editor. It shows checking/warming states during resolution and retains the existing private/noindex/owner-only behavior.
