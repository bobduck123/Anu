# Presence Agent OS — File Map

This folder contains the launch operating system for Presence and adjacent revenue/proof work.

## Core files

- `AGENTS.md` — root repo instruction file for coding agents.
- `SUBAGENT_ROUTING.md` — who/what should do each kind of task.
- `TASK_SIZING.md` — classify complexity, ambiguity, and blast radius.
- `NO_MERGE_REVIEW.md` — mandatory review checklist.
- `PRESENCE_CANON.md` — living project truth.
- `PLANS.md` — complex work/multi-hour execution plan format.

## Launch and proof files

- `PROOF_LIBRARY.md` — prior delivery inventory and case studies.
- `PRESENCE_MIGRATION.md` — migration process for existing presences.
- `LAUNCH_OPERATING_SYSTEM.md` — 30/90/12-month execution map.
- `CASE_STUDY_TEMPLATE.md` — standard case study format.

## Revenue and operations

- `REVENUE_PIPELINE.md` — offers, funnel, outreach, proposals.
- `PROFILER_FBI_CANON.md` — safe extraction of Profiler/FBi proof.
- `DAILY_OPERATOR.md` — daily/weekly hyperdrive cadence.
- `WORK_ORDER_TEMPLATE.md` — standard task brief.
- `SECURITY_AND_PRIVACY.md` — sensitive-data and public-claim rules.

## How to use

1. Use the confirmed npm commands in `AGENTS.md`; do not invent lint or unit-test scripts that are absent from `package.json`.
2. Fill `PROOF_LIBRARY.md` with all existing presences and delivered work.
3. Run a daily command pass using `DAILY_OPERATOR.md`.
4. Create work orders from `WORK_ORDER_TEMPLATE.md`.
5. For code tasks, follow `TASK_SIZING.md` and `SUBAGENT_ROUTING.md`.
6. For L/XL work, create an ExecPlan using `PLANS.md`.
7. Before merge or launch, apply `NO_MERGE_REVIEW.md`.
8. After shipping, update `PRESENCE_CANON.md` and `PROOF_LIBRARY.md`.
