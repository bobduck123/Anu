# Presence Studio V3 decision log

These decisions are binding across the Studio V3 specification set. They resolve the planning pass without pretending the prototype already exists. A future implementation may change a decision only through a scoped, reviewed update to this log and the affected specifications.

## Decision table

| ID | Decision | Reason | Consequence |
| --- | --- | --- | --- |
| D01 | Studio V3 is the replacement client-facing shell over the existing Studio V2 renderer, composition, projection, draft, preview, and publish boundaries. | V2 has technical proof; the owner experience, not the underlying engine, is the marketability gap. | No backend or renderer rewrite in the first slice. V2 remains an internal engine, not client vocabulary. |
| D02 | Enter V3 through the existing `/studio/[id]/editor` route behind a default-off, local/test, BBB-only pilot gate. Do not create a permanent `/editor-v3` route. | A parallel route would fragment evidence and become a second editor. | The route chooses one shell. Legacy V2 remains the fallback until the V3 gate is deliberately enabled. |
| D03 | The full-screen V3 shell branches outside the legacy `StudioShell` navigation and reduces top-level IA to Edit Presence, Library, Visitors, and Settings. | The current multi-tab cockpit is the product problem. | Existing owner pages can remain behind the four client areas; operator/debug fields stay hidden. |
| D04 | The live editor canvas uses `PresenceStudioV2PublicRoom`, including its BBB-specific rendering branch. | A generic editor canvas can diverge from what visitors see. | The optional editor bridge has a normative synchronous intent/result contract. Its branch runs inside the native BBB callbacks before focus/navigation side effects, consumes direct/window navigation and unsupported/no-ID chrome, cancels or rechecks pending focus, and routes only editor selection intent. **Open link** is deliberate and **Test as visitor** is a reversible in-memory bridge-free mode. Server-backed Visitor Preview remains unavailable until the atomic draft-save gate passes. With no bridge, public markup, focus, events, behavior, and output remain unchanged. |
| D05 | The client view model is Presence, Room, Piece, Collection, and Look. A Room compiles to a V2 chamber; a placed Piece compiles to a V2 room object. | Owner language should describe a digital home, not renderer payloads. | Backend Node/room/object terminology remains an adapter detail and is not shown to clients. |
| D06 | Works and Collections from the owner Library APIs are canonical content. V2 room objects are compiled placement snapshots, not canonical Works. | Owners must add/edit content once and arrange it in many presentations. | Placement records carry source references; Library deletion and room removal have different semantics. |
| D07 | Editor-local source references are explicit: `work:<id>`, `collection:<id>`, or `legacy-object:<id>`. Compiled visitor object IDs are opaque, deterministic, and unique per Room and placed Piece; raw source refs never enter public projection. | Stable identity is required for deduplication, reload, restore, and selection without leaking editor provenance. | Collection placement expands deterministically, skips duplicates, and reports incompatible items to a visible shelf; nothing is silently lost. Opaque IDs are not an authorization boundary. Browser-local provenance qualifies only for the exact loaded base; durable server-side placement/source identity is a pre-market gate. |
| D08 | Compilation starts from the exact latest overview draft config, falling back to current published config only when no draft exists, and changes only the normative compiler-owned paths/subtrees while preserving every unowned value. | Rebuilding from a partial selected node or treating numeric version as concurrency authority risks destructive omission and lost updates. | The ten-field logical comparison is built from defined wire JSON, with base `schema_version` injected unchanged; the current nine-field POST transport intentionally omits only that unsupported field and must retain every other top-level field after serialization. The stable stored-semantic fingerprint strips only owner-GET-added `url` and `preview_expires_at` on validated `private_draft` media references; every other value remains authoritative. Current recursive-merge `PATCH` is forbidden. The existing POST may be characterised only against a disposable single-writer local/test draft and never qualifies product Save. V3 never publishes or auto-rebases. |
| D09 | Prototype-only V3 state—layer locks, named Looks, savepoints, staged before/after state, and UI preferences—is browser-local in separate Presence and Room envelopes under an opaque authenticated-owner partition, scoped to the complete immutable base identity plus canonical full-base fingerprint. | The current editable-config serializer may expose unknown metadata and browser storage is shared across logins on one origin. | No media blobs, secrets, raw owner IDs, or full payload snapshots are stored. Authoritative mismatch is quarantined/discarded; logout/account switch clears the prior partition and cleanup uncertainty disables local storage. `updated_at` is diagnostic only. Durable sync is deferred to an approved contract. |
| D10 | The six customization layers are Presence Look, Room Style, Collection Presentation, Piece Treatment, Motion/Atmosphere, and Navigation/Journey. Override precedence is piece > collection > room > Presence. | Owners need power with predictable inheritance. | A preset changes only unlocked layers; overrides and provenance remain visible and editable. |
| D11 | The first Looks are Soft Editorial, Nocturnal Gallery, and Zine Archive. The first Room Styles are Threshold Portal, Gallery Wall, and Film Strip / Selected Works. | These are the smallest set that can prove material differences in atmosphere, composition, density, treatment, motion, and journey. | Threshold Portal and Gallery Wall adapt proven layouts. Film Strip is the single smallest new presentation adapter. Zine Board is only a re-reviewed fallback candidate if Film Strip proves unsafe; it is not silently substituted. Palette-only variants do not satisfy acceptance. |
| D12 | Image treatment, motion intensity, background, typography, and CTA styling update instantly. Room style, navigation structure, collection presentation, and transformation presets use staged preview/apply. | Fast reversible feedback builds confidence; structural changes need an explicit boundary. | Before/after compares the staged baseline with the staged result. Apply commits the browser-local working model and savepoint. Product **Save Draft** remains disabled until D22's atomic server gate passes; the current POST is not a concurrency boundary. Cancel restores the baseline. |
| D13 | Named Looks store normalized editable layer values/recommendations and provenance—not screenshots, lock state, or placement/order. Media may be retained only as an opaque stable owner-authorised asset ID; URLs, blobs, copied media, and private-draft refs are forbidden and media is omitted when no safe ID exists. Savepoints store structural/style references, placement order, locks, and required restore metadata. | Saved states must be real, editable, compact, and restorable without conflating restyling with structural rollback or leaking media references. | Destination locks remain independent when a named Look is applied. A transformation creates a savepoint before changing anything and produces a change/incompatibility summary. |
| D14 | Studio is mostly modeless: select an editable thing to act on it. The primary controls are a floating action bar and expandable bottom sheet. Simple mode is the client default; Advanced Creative is explicit opt-in. | The editor should feel spatial and touch-first without becoming a breakable design tool. | Simple mode exposes guided zones, reordering, feature state, safe sizes, crop/focus, and bounded motion. Advanced adds safe transforms, depth, and per-piece overrides. Operator/debug mode remains hidden. |
| D15 | The prototype uses current V2 pixel-transform behavior with safe bounds and snapping. A normalized responsive positioning contract is required before market launch or V3.1. | Replacing positioning in the first slice would broaden architecture, but pixel-only storage is not a market-ready cross-device contract. | The prototype proves interaction and compilation, not arbitrary freeform layout or final responsive fidelity. |
| D16 | Permanent Work deletion is available only in Library after confirmation that lists affected Collections and Room placements. Remove from Room and remove from Collection never delete the Work. Entry/last Room and required CTA protections are mandatory. | Direct manipulation must not create destructive ambiguity or broken visitor journeys. | Delete, remove, hide, and unfeature are separate commands with explicit outcomes and undo or recovery where feasible. |
| D17 | Milestone 1 proves an understandable, safe, live-feeling draft editor. Milestone 2 proves a radical BBB → GGM-like identity → BBB transformation with exact restoration inside the editor. | Comprehension and safety must be proven before transformative breadth. | GGM supplies private/local comparison qualities only. It is never published, routed, or used in public screenshots. |
| D18 | Current backend/content semantics requiring later cleanup are documented, not redesigned here: multimedia type coverage, richer media fields, Collection membership/cardinality and reordering, Room-placement relations, durable V3 state, and responsive transforms. | The first slice must stay on the proven system while being honest about market-launch debt. | Each cleanup requires a separate approved work order; none blocks the docs pass, while specified pre-market gates may block general availability. |
| D19 | Review & Publish stays visible as a safety destination, but actual publication occurs only inside the existing final checklist/confirmation and is absent from the prototype test path. | Owners should understand the lifecycle without accidental public mutation. | Tests assert zero publish requests and public BBB route invariance. Publish, deployment, and hosted mutation remain human gates. |
| D20 | The prototype has two explicit zero-write gates: P0 seam proof and P1 style proof. P0 proves the actual-renderer shell, selection/action bar/sheet, one Piece and Collection placement, one visibly changed Look, one lock, and named restore. P1 proves all three Looks, all three Room Styles including Film Strip, structural savepoint/compare, and independent aesthetic review. | A single “prototype” label previously mixed the smallest seam proof with the full first-market style claim. | P0 and P1 pass only with draft writes disabled. Milestone 1 may add broader local owner workflow, but runtime draft save/reload and server Visitor Preview remain a blocked subgate until D22 passes. |
| D21 | The canonical V3 test manifest is one pure `compiler.test.ts` plus BBB prototype, public-invariance, and mobile-accessibility Playwright specs. Every acceptance scenario records its first required gate and maps to one of those files. | Conflicting file names and untagged scenarios invite duplicate harnesses and accidental milestone expansion. | Pure tests use `npx.cmd tsx --test`; implementation stops for approval if that would require an unapproved download rather than adding a dependency silently. |
| D22 | Product/runtime draft replacement requires a separately approved server-side atomic precondition that checks the expected existing-draft identity/revision and stable stored-semantic fingerprint in the same transaction as replacement. | The current client refetch plus current POST has a time-of-check/time-of-use race, may create a missing draft, mutates private-media assignment state, and can detect a conflict only after mutation. | Until that contract exists and passes independent review, **Save Draft** and server Visitor Preview stay disabled. Missing/mismatched drafts must conflict before config or media mutation; media effects must roll back with the transaction. Refetch after POST is verification, not concurrency protection. This specification does not authorize the backend change. |

## First-slice state boundary

The prototype may keep a compact local envelope similar in concept to:

```text
opaque authenticated-owner partition
  + complete immutable base identity
  + canonical full-base fingerprint

Presence envelope
  Presence/global layers, locks, navigation, and staging
  named Look definitions and validated opaque asset IDs
  structural savepoint index
  simple/advanced preference

Room envelope
  Presence + Room scope
  Room layers, locks, placements, shelf/provenance, and staging
  Room savepoint fragments
```

This is an architectural boundary, not a final schema. It must not contain media bytes, auth data, owner/control-plane fields, raw public payloads, or private content not already authorized for that owner. The implementation specification owns the exact type and storage-key version.

## Superseded interpretation

The 2026-07-20 canon decision to keep Studio V2 as the single canonical editor remains true for the engine and eligible-Room path, but no longer means the existing V2 cockpit is the long-term client surface. V3 replaces that presentation layer in a gated sequence and compiles back into V2.

## Deferred decisions that do not block the prototype

- The durable server-side V3 metadata contract and migration strategy.
- Server-enforced atomic existing-draft preconditions and the separately qualified first-draft creation path. This does not block P0/P1 or local-only M1 work, but it blocks product Save Draft and server Visitor Preview.
- The normalized cross-breakpoint placement coordinate system.
- General-availability eligibility beyond the private BBB pilot.
- The full Room Style library after the first three proof styles.
- Whether later transition and entrance animation ships in V3.1 or a separately scoped effects slice.

Each deferred item has a milestone or risk gate in the specification set. None authorizes implementation beyond the prototype slice.
